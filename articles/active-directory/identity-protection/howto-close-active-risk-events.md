# <a name="how-to-close-active-risk-events"></a>Практическое руководство. Закрытие активных событий риска

С помощью [событий риска](../reports-monitoring/concept-risk-events.md) Azure Active Directory обнаруживает индикаторы потенциально скомпрометированных учетных записей пользователей. Администраторам нужно закрывать все события риска, чтобы затронутые пользователи больше не подвергались риску.

В этой статье описаны дополнительные варианты закрытия активных событий риска.

## <a name="options-to-close-risk-events"></a>Варианты закрытия событий риска 

У события риска может быть состояние **Активно** или **Закрыто**. Все активные события риска участвуют в вычислении значения, называемого уровнем риска для пользователя. Уровень риска для пользователя является индикатором (низким, средним, высоким) вероятности того, что учетная запись была скомпрометирована. 

Для закрытия активных событий риска есть такие варианты:

- Требование сброса пароля с помощью политики риска для пользователя

- Сброс паролей вручную
 
- Закрытие всех событий риска 

- Закрытие отдельных событий риска вручную



## <a name="require-password-reset-with-a-user-risk-policy"></a>Требование сброса пароля с помощью политики риска для пользователя

Настроив [политику условного доступа риска для пользователя](howto-user-risk-policy.md), можно настроить требование менять пароль, если был автоматически обнаружен определенный уровень риска для пользователя. 

![Сброс пароля](./media/howto-close-active-risk-events/13.png)

При сбросе пароля закрываются все активные события риска для соответствующего пользователя, и идентификатор снова считается безопасным. Использование политики риска — предпочтительный способ закрытия активных событий риска, так как этот метод автоматизирован. Взаимодействие между затронутым пользователем и службой поддержки или администратором не требуется.

Но использовать политику риска для пользователя не всегда возможно. Это касается, например, следующих случаев:

- пользователи, не зарегистрированные для многофакторной проверки подлинности (MFA);
- пользователи с активными событиями риска, которые были удалены;
- когда исследование показывает, что выявленные события риска произошли в результате действий полномочного пользователя.


## <a name="manual-password-reset"></a>Сброс паролей вручную

Если требование сброса пароля с помощью политики риска для пользователя установить невозможно, можно обеспечить закрытие всех событий риска для пользователя, сбросив пароль вручную.

![Сброс пароля](./media/howto-close-active-risk-events/04.png)


Соответствующее диалоговое окно предлагает два метода для сброса пароля:

![Сброс пароля](./media/howto-close-active-risk-events/05.png)


**Создание временного пароля** — создав временный пароль, можно немедленно вернуть удостоверение в безопасное состояние. Этот способ требует взаимодействия с затронутыми пользователями, так как им нужно знать временный пароль. Можно, например, отправить новый временный пароль по альтернативному адресу электронной почты пользователя или его руководителя. Так как пароль временный, при следующем входе пользователю будет предложено изменить его.


**Требование от пользователя сбросить пароль** — требование сброса пароля от пользователей позволяет включить самостоятельное восстановление без обращения в службу технической поддержки или к администратору. Как и в случае с политикой риска для пользователя, этот метод применяется только к пользователям, которые зарегистрированы для многофакторной проверки подлинности (MFA). Для пользователей, которые еще не зарегистрированы для MFA, этот параметр недоступен.


## <a name="dismiss-all-risk-events"></a>Закрытие всех событий риска

Если сброс пароля вам не подходит, можно также закрыть все события риска сразу. 

![Сброс пароля](./media/howto-close-active-risk-events/03.png)

При нажатии кнопки **Dismiss all events**(Закрыть все события) закрываются все события и соответствующий пользователь больше не находится под угрозой. Но так как этот метод не оказывает влияния на существующий пароль, он не возвращает связанный идентификатор в безопасное состояние. Предпочтительный сценарий использования этого метода — для удаленного пользователя с активными событиями риска. 


## <a name="close-individual-risk-events-manually"></a>Закрытие отдельных событий риска вручную

Отдельные события риска можно закрывать вручную. Закрывая события риска вручную, можно понизить уровень риска для пользователя. Как правило, события риска закрываются вручную в ответ на связанное расследование. Например, когда при общении с пользователем выясняется, что активное событие риска больше не требуется. 
 
При закрытии событий риска вручную можно изменить статус события с помощью любого из следующих действий на выбор:

![Действия](./media/howto-close-active-risk-events/06.png)

- **Разрешить** — если после исследования события риска вы выполнили соответствующее действие по исправлению вне службы защиты идентификации и считаете, что событие следует закрыть, пометьте его как разрешенное. Разрешенным событиям риска будет присвоено состояние "Закрыто", и они больше не будут учитываться при расчете риска для пользователя.

- **Пометить как ложное срабатывание** — иногда в результате исследования событий риска может обнаружиться, что событие помечено по ошибке. Количество таких вхождений можно уменьшить, пометив события риска как "Ложное срабатывание". Это позволит усовершенствовать классификацию таких событий при использовании алгоритмов машинного обучения. Состояние событий с ложным срабатыванием изменяется на "Закрыто", и они больше не учитываются при оценке риска для пользователя.

- **Пропустить** — если вы не предпринимали действий по исправлению, но хотите удалить события риска из списка активных, можно пометить их, щелкнув "Пропустить", чтобы изменить состояние на "Закрыто". Пропущенные события не влияют на риск для пользователя. Эту возможность следует использовать только при чрезвычайных обстоятельствах.

- **Повторная активация** — события риска, закрытые вручную (если вы щелкнули "Разрешить", "Ложное срабатывание" или "Пропустить"), можно активировать снова, установив для них состояние "Активно". Повторно активированные события рисков учитываются при расчете уровня риска для пользователя. События рисков, закрытые вследствие исправления (например, безопасного сброса пароля), нельзя повторно активировать.
