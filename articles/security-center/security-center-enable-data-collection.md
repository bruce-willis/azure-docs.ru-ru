# <a name="data-collection-in-azure-security-center"></a>Сбор данных в центре безопасности Azure
Центр безопасности собирает данные c виртуальных машин Azure и c компьютеров, не относящихся к Azure, чтобы отслеживать уязвимости и угрозы безопасности. Для сбора данных используется агент Microsoft Monitoring Agent, который считывает различные конфигурации, связанные с безопасностью, и журналы событий с компьютера, а также копирует данные в рабочую область для анализа. К примерам таких данных относятся тип и версия операционной системы, журналы операционной системы (журналы событий Windows), выполняющиеся процессы, имя компьютера, IP-адреса и имя вошедшего пользователя. Этот агент также копирует файлы аварийных дампов в рабочую область.

Сбор данных необходим, чтобы обнаруживать недостающие обновления и неправильно настроенные параметры безопасности ОС, узнавать, включена ли защита конечных точек, а также определять работоспособность и обнаруживать угрозы. 

В этой статье описывается, как установить Microsoft Monitoring Agent и настроить рабочую область Log Analytics для хранения собранных данных. Чтобы включить сбор данных, необходимо выполнить обе операции. 

> [!NOTE]
> - Сбор данных необходим только для ресурсов вычислительной среды (виртуальных машин и компьютеров, не относящихся к Azure). Вы можете пользоваться центром безопасности Azure даже без подготовки агентов. Однако защита будет ограничена, а перечисленные выше возможности не будут поддерживаться.  
> - Список поддерживаемых платформ представлен в статье [Поддерживаемые платформы в центре безопасности Azure](security-center-os-coverage.md).
> - Сбор данных для масштабируемого набора виртуальных машин в настоящее время не поддерживается.


## <a name="enable-automatic-provisioning-of-microsoft-monitoring-agent"></a>Включение автоматической подготовки агента Microsoft Monitoring Agent     
Для сбора данных с компьютеров должен быть установлен агент Microsoft Monitoring Agent.  Устанавливать агент можно автоматически (это рекомендуется) или вручную.  

>[!NOTE]
> Автоматическая подготовка по умолчанию отключена. Чтобы центр безопасности использовал автоматическую подготовку по умолчанию, выберите вариант **Вкл**.
>

Если автоматическая подготовка включена, центр безопасности подготавливает агент Microsoft Monitoring Agent к работе на всех поддерживаемых и новых виртуальных машинах Azure. Мы настоятельно рекомендуем применять автоматическую подготовку, но вы также можете установить агент вручную. Дополнительные сведения см. в разделе [Включение расширения виртуальной машины Log Analytics](#manualagent).



Чтобы включить автоматическую подготовку Microsoft Monitoring Agent, сделайте следующее:
1. В главном меню центра безопасности выберите **Политика безопасности**.
2. Выберите подписку.

  ![Выберите подписку.][7]

3. В разделе **Политика безопасности** выберите **Сбор данных**.
4. В разделе **Автоматическая подготовка** выберите **Включить**, чтобы включить автоматическую подготовку.
5. Щелкните **Сохранить**.

  ![Включение автоматической подготовки][1]

>[!NOTE]
> - Инструкции по подготовке уже существующей установки см. в разделе [Автоматическая подготовка уже существующей установки агента](#preexisting).
> - Инструкции по подготовке вручную см. в разделе [Установка расширения Microsoft Monitoring Agent вручную](#manualagent).
> - Инструкции по отключению автоматической подготовки см. в разделе [Отключение автоматической подготовки](#offprovisioning).
>


## <a name="workspace-configuration"></a>Конфигурация рабочей области
Данные, собранные центром безопасности, хранятся в рабочих областях Log Analytics.  Вы можете настроить хранение данных, собранных с виртуальных машин Azure, в рабочих областях, созданных центром безопасности, или в созданной вами рабочей области. 

Конфигурация рабочей области задается для каждой подписки отдельно, а несколько подписок могут использовать одну и ту же рабочую область.

### <a name="using-a-workspace-created-by-security-center"></a>Использование рабочей области, созданной центром безопасности

Центр безопасности может автоматически создать стандартную рабочую область для хранения данных. 

Чтобы выбрать рабочую область, созданную центром безопасности:

1.  В разделе **Конфигурация рабочей области по умолчанию** выберите параметр "Использовать рабочие области, созданные центром безопасности".
   ![Выбор ценовой категории][10] 

2. Выберите команду **Сохранить**.<br>
    Центр безопасности создает группу ресурсов и рабочую область по умолчанию в этом географическом расположении, а затем подключает агент к этой рабочей области. Соглашение об именовании для рабочей области и группы ресурсов:<br>
**Рабочая область: DefaultWorkspace-[subscription-ID]-[geo]<br> Группа ресурсов: DefaultResouceGroup-[geo]**

   Если подписка содержит виртуальные машины из нескольких географических расположений, центр безопасности создает несколько рабочих областей. Они создаются для соблюдения правил конфиденциальности данных.
-   Центр безопасности автоматически включит решение центра безопасности в рабочей области согласно ценовой категории, заданной для подписки. 

> [!NOTE]
> За использование Log Analytics в рабочих областях, созданных центром безопасности, не взимается плата. Ценовая категория Log Analytics для рабочих областей, созданных центром безопасности, не влияет на выставление счетов за его использование. Счета за использование центра безопасности всегда выставляются в зависимости от его политики безопасности и решений, установленных в рабочей области. Для уровня "Бесплатный" центр безопасности включает решение *SecurityCenterFree* в рабочей области по умолчанию. Для уровня "Стандартный" центр безопасности включает решение *Security* в рабочей области по умолчанию.

Дополнительные сведения см. на странице с [ценами на центр безопасности](https://azure.microsoft.com/pricing/details/security-center/).

Дополнительные сведения об имеющихся учетных записях Log Analytics см. в разделе [Существующие клиенты Log Analytics](security-center-faq.md#existingloganalyticscust).

### <a name="using-an-existing-workspace"></a>Использование имеющейся рабочей области

Если у вас уже есть рабочая область Log Analytics, вы можете продолжать использовать ее.

Чтобы использовать имеющуюся рабочую область Log Analytics, вам потребуются разрешение на чтение и запись в этой рабочей области.

> [!NOTE]
> Решения, включенные в имеющейся рабочей области, будут применяться к подключенным к ним виртуальным машинам Azure. В случае платных решений за это может взиматься дополнительная плата. Из соображений конфиденциальности данных убедитесь, что выбранная рабочая область находится в нужном географическом регионе.
>

Чтобы выбрать имеющуюся рабочую область Log Analytics, сделайте следующее:

1. В разделе **Конфигурация рабочей области по умолчанию** выберите **Использовать другую рабочую область**.

   ![Выбор имеющейся рабочей области][2]

2. В раскрывающемся меню выберите рабочую область для хранения собранных данных.

  > [!NOTE]
  > В этом раскрывающемся меню доступны все рабочие области во всех ваших подписках. Ознакомьтесь с разделом [Выбор рабочих областей между подписками](security-center-enable-data-collection.md#cross-subscription-workspace-selection), чтобы получить дополнительную информацию. Вам необходимо разрешение на доступ к рабочей области.
  >
  >

3. Щелкните **Сохранить**.
4. Если нажать **Сохранить**, появится запрос на изменение настроек отслеживаемых виртуальных машин, которые ранее были подключены к рабочей области по умолчанию.

   - Выберите **Нет**, если нужно, чтобы параметры новой рабочей области применялись только к новым виртуальным машинам. Параметры новой рабочей области применяются только к новым установленным агентам (недавно обнаруженные виртуальные машины, на которых не установлен агент Microsoft Monitoring Agent).
   - Выберите **Да**, если нужно, чтобы параметры новой рабочей области применялись ко всем виртуальным машинам. Кроме того, каждая виртуальная машина, подключенная к рабочей области, созданной центром безопасности, переподключается к новой целевой рабочей области.

   > [!NOTE]
   > Если вы выбрали "Да", не удаляйте рабочие области, созданные центром безопасности, пока все виртуальные машины не подключатся повторно к новой целевой рабочей области. Операция завершится сбоем, если удалить рабочую область слишком рано.
   >
   >

   - Выберите **Отменить**, чтобы отменить операцию.

     ![Выбор имеющейся рабочей области][3]

5. Выберите ценовую категорию для нужной рабочей области, в которой требуется настроить Microsoft Monitoring Agent. <br>Чтобы использовать имеющуюся рабочую область, задайте для нее ценовую категорию. При этом в рабочей области будет установлено решение центра безопасности, если его еще нет.

    a.  В главном меню центра безопасности выберите пункт **Политика безопасности**.
     
    b.  Выберите нужную рабочую область, в которой требуется подключить агент.
        ![Выбор рабочей области][8] c. Выберите ценовую категорию.
        ![Выбор ценовой категории][9] 
   
   >[!NOTE]
   >Если в рабочей области уже включено решение **Security** или **SecurityCenterFree**, ценовая категория будет задана автоматически. 

## <a name="cross-subscription-workspace-selection"></a>Выбор рабочих областей между подписками
При выборе рабочей области для хранения данных доступны все рабочие области во всех ваших подписках. Выбор рабочих областей между подписками позволяет собирать данные из виртуальных машин, работающих в разных подписках, и сохранять их в выбранной рабочей области. Это полезно, если в организации используется централизованная рабочая область, которую требуется использовать для сбора данных о безопасности. Дополнительные сведения о том, как управлять рабочими областями, см. в статье [Управление рабочими областями](https://docs.microsoft.com/azure/log-analytics/log-analytics-manage-access).


## <a name="data-collection-tier"></a>Уровень сбора данных
Центр безопасности может сократить количество событий, сохранив достаточное количество событий для исследования, аудита и обнаружения угроз. Вы можете выбрать необходимую политику фильтрации для подписок и рабочих областей из четырех наборов событий, которые будут собираться агентом.

- **Все события.** Для клиентов, которым необходимо убедиться, что собраны все события. Это уровень по умолчанию.
- **Общий.** Этот набор событий удовлетворяет потребности большинства клиентов и позволяет просмотреть весь журнал аудита.
- **Минимальный.** Меньший набор событий для клиентов, которые хотят сократить количество событий.
- **Отсутствует.** Отключение сбора событий безопасности из журналов безопасности и App Locker. На панелях мониторинга клиентов, выбравших этот вариант, будут находиться только журналы брандмауэра Windows и инструменты профилактической оценки, например антивредоносная программа, базовый план и обновление.

> [!NOTE]
> Эти наборы событий безопасности доступны только на уровне "Стандартный" центра безопасности. Дополнительные сведения о ценовых категориях центра безопасности см. на странице [Цены](security-center-pricing.md).
Эти наборы предназначены для стандартных сценариев. Выберите набор, который соответствует вашим требованиям, прежде чем внедрить его.
>
>

Для определения событий, которые будут принадлежать к наборам событий **Общий** и **Минимальный**, мы ознакомились с отзывами клиентов и отраслевыми стандартами, чтобы изучить нефильтрованную частоту каждого события и их использования. В процессе мы использовали приведенные ниже рекомендации.

- **Минимальный.** Убедитесь, что этот набор содержит только события, которые могут указывать на брешь в системе безопасности и важные события небольших объемов. Например, этот набор содержит относительно большой объем сведений об успешных и неудачных попытках входа пользователя (идентификаторы события 4624 и 4625), однако не содержит сведений о выходе, что очень важно для аудита и не важно для обнаружения. Большую часть данных этого набора занимают события входа и создания процессов (идентификатор событий 4688).
- **Общий.** В этом наборе предоставляется полный журнал аудита пользователя. Например, этот набор содержит данные о входе и выходе пользователя (идентификатор события 4634). Мы добавили действия аудита, такие как изменение группы безопасности, операции основного контроллера домена Kerberos и другие события, рекомендованные отраслевыми организациями.

События небольших объемов были включены в набор "Общий". Основной причиной для его выбора среди всех событий является возможность уменьшить объем и оставить определенные события нефильтрованными.

### Автоматическая подготовка уже существующей установки агента <a name="preexisting"></a> 

В описанных ниже вариантах использования указано, как автоматическая подготовка работает в тех случаях, когда агент или расширение уже установлены. 

- Microsoft Monitoring Agent установлен на компьютере, но не в качестве расширения<br>
Если агент Microsoft Monitoring Agent установлен непосредственно на виртуальной машине (не как расширение Azure), центр безопасности не устанавливает его. Вы можете включить автоматическую подготовку и выбрать соответствующую рабочую область пользователя в конфигурации автоматической подготовки в центре безопасности. Если выбрать рабочую область, к которой уже подключена виртуальная машина, имеющийся агент будет помещен в расширение Microsoft Monitoring Agent. 

> [!NOTE]
> Если установлен агент SCOM версии 2012, **не** включайте автоматическую подготовку. 

Дополнительные сведения см. в разделе [Что произойдет, если прямой агент SCOM или OMS уже установлен на моей виртуальной машине?](security-center-faq.md#scomomsinstalled)

-   Расширение виртуальной машины уже установлено<br>
    - Центр безопасности поддерживает установленные расширения и не переопределяет имеющиеся подключения. Центр безопасности хранит данные о безопасности с виртуальной машины в уже подключенной рабочей области и обеспечивает защиту, соответствующую включенным в ней решениям.   
    - Чтобы узнать, в какую рабочую область отправляет данные имеющееся расширение, выполните тест для [проверки соединения с центром безопасности Azure](https://blogs.technet.microsoft.com/yuridiogenes/2017/10/13/validating-connectivity-with-azure-security-center/). Вы также можете открыть Log Analytics, выбрать рабочую область и виртуальную машину, а затем проверить соединение Microsoft Monitoring Agent. 
    - Если в вашей среде на клиентских рабочих станциях установлен агент Microsoft Monitoring Agent, отправляющий отчеты в имеющуюся рабочую область Log Analytics, просмотрите список [операционных систем, поддерживаемых центром безопасности Azure](security-center-os-coverage.md), чтобы убедиться, что ваша операционная система поддерживается, и ознакомьтесь с дополнительными сведениями из раздела [Существующие клиенты Log Analytics](security-center-faq.md#existingloganalyticscust).
 
### Отключение автоматической подготовки <a name="offprovisioning"></a>
Отключить автоматическую подготовку ресурсов можно в любое время, отключив этот параметр в политике безопасности. 


1. Вернитесь к основному меню центра безопасности и выберите "Политика безопасности".
2. Выберите подписку, в которой нужно отключить автоматическую подготовку.
3. В колонке **Security policy – Data Collection** (Политика безопасности — сбор данных) в разделе **Автоматическая подготовка** выберите **Отключить**.
4. Щелкните **Сохранить**.

  ![Отключение автоматической подготовки][6]

Если автоматическая подготовка отключена, раздел конфигурации рабочей области по умолчанию не отображается.

Если отключить автоматическую подготовку:
-   агенты не будут подготавливаться на новых виртуальных машинах;
-   центр безопасности перестанет собирать данные из рабочей области по умолчанию.
 
> [!NOTE]
>  При отключении автоматической подготовки агент Microsoft Monitoring Agent не удаляется с виртуальных машин Azure, на которых он был подготовлен. Сведения о том, как удалить расширение OMS, см. в разделе [Как удалить расширения OMS, установленные центром безопасности](security-center-faq.md#remove-oms).
>
    
## Подготовка агента вручную <a name="manualagent"></a>
 
Установить Microsoft Monitoring Agent вручную можно несколькими способами. При установке вручную обязательно отключите автоматическую подготовку.

### <a name="operations-management-suite-vm-extension-deployment"></a>Развертывание расширения виртуальной машины Operations Management Suite 

Вы можете вручную установить Microsoft Monitoring Agent, чтобы центр безопасности мог собирать данные с виртуальных машин, а также предоставлять рекомендации и предупреждения.
1.  Выберите для параметра "Автоматическая подготовка" значение "ВЫКЛ".
2.  Создайте рабочую область и выберите ценовую категорию для установки Microsoft Monitoring Agent:

    a.  В главном меню центра безопасности выберите пункт **Политика безопасности**.
     
    b.  Выберите рабочую область, в которой требуется подключить агент. Убедитесь, что рабочая область находится в той же подписке, которая используется в центре безопасности, а у вас есть разрешения на чтение и запись в этой рабочей области.
        ![Выбор рабочей области][8]
3. Выберите ценовую категорию.
   ![Выбор ценовой категории][9] 
   >[!NOTE]
   >Если в рабочей области уже включено решение **Security** или **SecurityCenterFree**, ценовая категория будет задана автоматически. 
   > 

4.  Если вы хотите развернуть агенты на новых виртуальных машинах с помощью шаблона Resource Manager, установите расширение виртуальной машины OMS:

    a.  [Установка расширения виртуальной машины OMS для Windows](../virtual-machines/extensions/oms-windows.md)
    
    b.  [Установка расширения виртуальной машины OMS для Linux](../virtual-machines/extensions/oms-linux.md)
5.  Чтобы развернуть расширения на имеющихся виртуальных машинах, следуйте инструкциям из статьи [Сбор данных о виртуальных машинах Azure](../log-analytics/log-analytics-quick-collect-azurevm.md).

  > [!NOTE]
  > Действия из раздела **Сбор данных о событиях и производительности** выполнять необязательно.
  >
6. Чтобы развернуть расширение с помощью PowerShell, используйте следующий пример PowerShell:
    1.  Перейдите к **Log Analytics** и выберите **Дополнительные параметры**.
    
        ![Настройка Log Analytics][11]

    2. Скопируйте значения из полей **WorkspaceID** и **Первичный ключ**.
  
       ![Копирование значений][12]

    3. Заполните общедоступную и частную конфигурации следующими значениями:
     
            $PublicConf = '{
                "workspaceId": "WorkspaceID value",
                "MultipleConnectistopOnons": true
            }' 
 
            $PrivateConf = '{
                "workspaceKey": "<Primary key value>”
            }' 

      - При установке на виртуальной машине Windows:
        
             Set-AzureRmVMExtension -ResourceGroupName $vm.ResourceGroupName -VMName $vm.Name -Name "MicrosoftMonitoringAgent" -Publisher "Microsoft.EnterpriseCloud.Monitoring" -ExtensionType "MicrosoftMonitoringAgent" -TypeHandlerVersion '1.0' -Location $vm.Location -Settingstring $PublicConf -ProtectedSettingString $PrivateConf -ForceRerun True 
    
       - При установке на виртуальной машине Linux:
        
             Set-AzureRmVMExtension -ResourceGroupName $vm1.ResourceGroupName -VMName $vm1.Name -Name "OmsAgentForLinux" -Publisher "Microsoft.EnterpriseCloud.Monitoring" -ExtensionType "OmsAgentForLinux" -TypeHandlerVersion '1.0' -Location $vm.Location -Settingstring $PublicConf -ProtectedSettingString $PrivateConf -ForceRerun True`
