# <a name="define-test-and-use-workflows"></a>Определение, тестирование и использование рабочих процессов

Конструктор рабочих процессов Azure Content Moderator и интерфейсы API можно использовать для определения пользовательских рабочих процессов и порогов на основе политик содержимого.

Рабочие процессы "подключаются" к API Content Moderator с помощью соединителей. Можно использовать другие интерфейсы API, если для них доступен соединитель. В данном примере используется соединитель Content Moderator, который предоставляется по умолчанию.

## <a name="browse-to-the-workflows-section"></a>Переход к разделу "Workflows" (Рабочие процессы)

На вкладке **Settings** (Параметры) выберите **Workflows** (Рабочие процессы).

  ![Настройка рабочих процессов](images/2-workflows-0.png)

## <a name="start-a-new-workflow"></a>Запуск нового рабочего процесса

Выберите **Add Workflow** (Добавить рабочий процесс).

  ![Добавление рабочего процесса](images/2-workflows-1.png)

## <a name="assign-a-name-and-description"></a>Задание имени и описания

Задайте имя рабочего процесса, введите его описание и укажите, что обрабатывает рабочий процесс: изображения или текст.

  ![Имя и описание рабочего процесса](images/ocr-workflow-step-1.PNG)

## <a name="define-the-evaluation-criteria-condition"></a>Определение критериев оценки ("состояния")

На следующем снимке экрана показаны поля и условия If-Then-Else, выбранные для рабочих процессов. Выберите соединитель. В этом примере используется **Content Moderator**. Доступные параметры выходных данных зависят от выбранного соединителя.

  ![Определение условия рабочего процесса](images/ocr-workflow-step-2-condition.PNG)

После выбора соединителя и его выходных данных следует выбрать оператор и значение для условия.

## <a name="define-the-action-to-take"></a>Определение предпринимаемого действия

Выберите действие и условие, которое должно быть выполнено. Следующий пример создает данные проверки изображения, назначает тег `a`и выделяет его для отображенного условия. Кроме того, можно объединить несколько условий, чтобы получить нужные результаты. При необходимости добавьте альтернативный путь (Else).

  ![Определение действия рабочего процесса](images/ocr-workflow-step-3-action.PNG)

## <a name="save-your-workflow"></a>Сохранение рабочего процесса

Наконец, сохраните рабочий процесс и запишите его имя. Оно нужно для запуска задания модерации с помощью API проверки.

## <a name="test-the-workflow"></a>Проверка рабочего процесса

Теперь, когда вы определили пользовательский рабочий процесс, проверьте его с помощью примера содержимого.

Нажмите соответствую кнопку **Execute Workflow** (Выполнить рабочий процесс).

  ![Проверка рабочего процесса](images/ocr-workflow-step-6-list.PNG)

### <a name="upload-a-file"></a>Отправка файла

Сохраните [пример изображения](https://moderatorsampleimages.blob.core.windows.net/samples/sample5.png) на локальный диск. Чтобы проверить рабочий процесс, выберите **Choose File(s)** (Выбор файлов) и передайте изображение.

  ![Передача файла изображения](images/ocr-workflow-step-7-upload.PNG)

### <a name="track-the-workflow"></a>Отслеживание рабочего процесса

Отслеживайте выполнение рабочего процесса.

  ![Отслеживание выполнения рабочего процесса](images/ocr-workflow-step-4-test.PNG)

### <a name="review-any-images-flagged-for-human-moderation"></a>Просмотр изображений, помеченных для пользовательской модерации

Чтобы просмотреть результат проверки изображения, перейдите на вкладку **Image** (Изображение) в разделе **Review** (Проверка).

  ![Проверка образов](images/ocr-sample-image-workflow1.PNG)

## <a name="next-steps"></a>Дополнительная информация 

Чтобы вызвать рабочий процесс из кода, используйте пользовательские рабочие процессы, описанные в кратком руководстве по запуску [`Job` из консоли API](../try-review-api-job.md) и [кратком руководстве по пакету SDK для .NET](../moderation-jobs-quickstart-dotnet.md).
